SEARCHSPLOIT - Exploit-DB Search Tool

Overview

searchsploit is a command-line tool to search the Exploit-DB database. It allows you to quickly find exploits for software vulnerabilities without requiring internet access (local mirror) or with online search.

Why It's Critical for eJPT:

  • Quickly find exploits for identified software/versions
  • Works offline (if database is synced)
  • Faster than browsing Exploit-DB website
  • Helps prioritize vulnerabilities by exploit availability
  • Essential workflow: nmap → identify version → searchsploit

Installation & Setup

CachyOS (Arch Linux)

sudo pacman -S exploitdb
searchsploit -h  # Verify
which searchsploit

Update Local Database

# Sync with latest exploits:
sudo searchsploit -u
# On first run, downloads ~50GB of exploits

Windows

  • Install via WSL with Arch/CachyOS
  • Or download from: https://www.exploit-db.com/

Common Use Cases

1. Search for Exploit by Software Name

searchsploit Apache 2.4.6
searchsploit Wordpress 5.0
searchsploit Tomcat 9.0

Output:

-------------------------------------------------- ---------------------------------
 Exploit Title                                    |  Path
-------------------------------------------------- ---------------------------------
Apache 2.4.6 - Buffer Overflow                   | linux/remote/24659.c
Apache 2.4.6 - Heap-based Buffer Overflow        | linux/remote/39639.rb
Apache mod_ssl 2.4.2 - Denial of Service         | linux/dos/16439.rb
-------------------------------------------------- ---------------------------------

2. Search by CVE ID

searchsploit CVE-2017-9822
searchsploit CVE-2019-1010317

3. Filter by Type

searchsploit -t Apache | grep "remote"  # Only remote exploits
searchsploit -t Apache | grep "local"   # Only local exploits

4. Exclude Terms

searchsploit Apache -e Denial  # Exclude Denial of Service
searchsploit --exclude="dos"

5. Copy Exploit to Current Directory

searchsploit -p 24659  # Shows path to exploit
searchsploit -p 24659 | tr -d '\n' | xargs -I {} cp {} .

Flag: - -p: Show full path to exploit

6. View Exploit Details

searchsploit -x 24659
# Opens exploit code in text viewer

7. Case-Insensitive Search

searchsploit -i apache  # Lowercase a

8. Update Database

sudo searchsploit -u

Practical eJPT Workflow

Complete Exploitation Path

# Step 1: Identify software version with nmap
nmap -sV 192.168.1.100 -p 80

# Output: Apache/2.4.6

# Step 2: Search for exploits
searchsploit Apache 2.4.6

# Step 3: Review available exploits
# Choose one that seems viable

# Step 4: Copy exploit to working directory
searchsploit -p 24659 | xargs -I {} cp {} .

# Step 5: Examine exploit code
cat 24659.c
# or
searchsploit -x 24659

# Step 6: Compile/run exploit
gcc 24659.c -o exploit
./exploit 192.168.1.100

Real-World Example

$ searchsploit Apache 2.4.6
  Exploit Title                                    |  Path
  ------------------------------------------------ | ---------
  Apache 2.4.6 - CVE-2013-6438 - Heap Overflow   | linux/remote/24659.c

$ searchsploit -p 24659
  Path: /usr/share/exploitdb/exploits/linux/remote/24659.c

$ cat /usr/share/exploitdb/exploits/linux/remote/24659.c
  // Exploit code here...

$ gcc 24659.c -o apache_exploit
$ ./apache_exploit 192.168.1.100
  [+] Attempting exploit...
  [+] Shell obtained!

Tips

  • Always verify CVE: Confirm the vulnerability applies to your target version
  • Check exploit type: Local vs Remote, DOS vs Code Execution
  • Read the code: Understand what exploit does before running
  • Keep database updated: sudo searchsploit -u regularly
  • Combine with nmap: Always identify versions first

Key Takeaway: searchsploit [software] [version] → find exploits → use Metasploit or run manually

CURL - HTTP Testing Tool

Overview

curl is a command-line tool for making HTTP/HTTPS requests. In penetration testing, it's used for manual web application testing, API testing, and HTTP reconnaissance.

Why It's Critical for eJPT:

  • Manual HTTP request crafting
  • Test for injection vulnerabilities
  • API endpoint testing
  • Data exfiltration
  • Simple server testing
  • Quick web reconnaissance

Installation & Setup

CachyOS (Arch Linux)

sudo pacman -S curl
curl --version  # Verify

Windows

  • Usually pre-installed or use WSL
  • Download: https://curl.se/download.html

Common Use Cases

1. Basic GET Request

curl http://target.com
curl -v http://target.com  # Verbose (see headers)

2. HEAD Request (Headers Only)

curl -I http://target.com

Output shows: - Server header - Content-Type - Set-Cookie - Other metadata

3. POST Request with Data

curl -X POST -d "username=admin&password=pass" http://target.com/login
curl -X POST --data "id=1" http://target.com/search

4. Send Custom Headers

curl -H "User-Agent: Mozilla/5.0" http://target.com
curl -H "Authorization: Bearer TOKEN" http://target.com/api

5. Authentication

curl -u username:password http://target.com
curl --digest -u user:pass http://target.com

6. Cookies

curl -b "session=abc123" http://target.com
curl -c cookies.txt http://target.com  # Save cookies
curl -b cookies.txt http://target.com  # Send saved cookies

7. Follow Redirects

curl -L http://target.com

8. Test SQL Injection

curl "http://target.com/page?id=1' OR '1'='1"
curl "http://target.com/page?id=1 UNION SELECT NULL--"

9. Upload File

curl -F "file=@shell.php" http://target.com/upload

10. Send JSON

curl -X POST -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"pass"}' \
  http://target.com/api/login

11. Proxy Request

curl -x http://proxy:8080 http://target.com
curl -x http://user:pass@proxy:8080 http://target.com

12. SSL/TLS Certificate Issues

curl -k https://target.com  # Skip certificate validation

13. Save Output

curl http://target.com -o output.html
curl http://target.com > output.html

14. Resume Download

curl -C - -o file.zip http://target.com/file.zip

15. Batch Testing

# Test multiple endpoints:
for endpoint in /admin /login /api/users /backdoor; do
  echo "Testing $endpoint:"
  curl -I http://target.com$endpoint
done

Practical eJPT Workflow

Test Web Application Manually

# Step 1: Basic page retrieval
curl -v http://target.com/page

# Step 2: Check headers for info
curl -I http://target.com

# Step 3: Test for SQL injection
curl "http://target.com/search?q=1' OR '1'='1"

# Step 4: Try authentication bypass
curl -X POST -d "username=admin&password=test" http://target.com/login

# Step 5: Send malicious input
curl "http://target.com/search?q=<script>alert('xss')</script>"

# Step 6: API enumeration
curl http://target.com/api/v1/users
curl http://target.com/api/v1/admin
curl http://target.com/api/v2/config

Real-World Example

# Test login form for SQL injection:
$ curl -v -X POST -d "user=admin' OR '1'='1&pass=test" \
  http://target.com/login

# Response shows 200 OK (login bypassed!)
# Next step: Upload shell or access admin panel

Tips

  • Always use -v: See complete HTTP conversation
  • Combine with grep: Extract specific response data
  • Save requests: Use -D for cookies, headers for later use
  • Test methodically: Don't randomly guess, follow logic
  • Use with Burp Suite: Burp is better for interactive testing, curl for automation

Key Takeaway: curl [options] URL → manual HTTP testing, simpler than Burp for quick tests
